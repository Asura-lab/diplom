\fref{fig:system_architecture}-т харуулсан архитектур нь дөрвөн үндсэн бүсээс бүрдэнэ.

\textbf{Хэрэглэгчийн хэсэг} (дээд тал): хэрэглэгчид React Native мобайл апп (Android · iOS)-ын тусламжтайгаар системд хандана. Хүсэлт DNS-ээр дамжин шууд сервер рүү чиглэгдэнэ.

\textbf{Сервер} (төв хэсэг): нэг Docker контейнер дотор Flask + Gunicorn (API сервер) болон \emph{фон процесс} (background worker) гэсэн хоёр процесс зэрэгцэн ажиллана. API сервер нь \texttt{@app.route} декораторт бүртгэсэн ~30 HTTP интерфэйсийг (endpoint) гаргадаг. Фон процесс нь thread-д суурилсан хуваарилагчаар (scheduler) дохио үүсгэх (60 сек), мэдээ шинэчлэх (30 мин), мэдэгдэл илгээх, хослолын шинжилгээ хийх даалгавруудыг гүйцэтгэдэг. Мөн GBDT (LightGBM + XGBoost + CatBoost) ансамблийн таамаглалыг (inference) тус процесс хариуцна. Хоёр процессыг \texttt{APP\_PROCESS\_ROLE} орчны хувьсагчаар (\texttt{api}\,/\,\texttt{worker}\,/\,\texttt{all}) тохируулна.

Серверийн доод хэсэгт өгөгдлийн давхарга байрладаг. \emph{Кэш сан} нь сонголтот Redis (хүсэлтийн хязгаарлалтад (rate limiting) хэрэглэгддэг ба байхгүй үед санах ойн нөөц шийдэлд (fallback) шилждэг). \emph{MongoDB} нь үндсэн агуулах болж системийн бүх төлөвийг (хэрэглэгчийн профайл, дохио, нэвтрэлтийн токен, push мэдэгдэл, мэдээний шинжилгээ, даалгаврын дараалал г.м.) цуглуулгуудад хадгална; цуглуулга бүрийн схемийг Хавсралт~Д-ээс үзнэ үү.

\textbf{Дотоод сервис} (зүүн тал, улаан): системийн дотоодоос ашиглах туслах сервисүүд — \emph{И-мэйл} нь Flask-Mail SMTP (Gmail) ашиглан баталгаажуулах болон нууц үг сэргээх кодыг илгээдэг; \emph{Даалгаврын дараалал} нь \texttt{analysis\_jobs} цуглуулга дахь LLM зах зээлийн шинжилгээний даалгаврын дараалал ба \texttt{job\_locks} (TTL индекстэй) хуваарилагдсан түгжээгээс бүрдэнэ; \emph{Хяналт} нь \texttt{/health} интерфэйс болон Fly.io лог-ыг хамардаг.

\textbf{Гуравдагч талын API} (баруун тал, ногоон): дөрвөн гадны API нь нэгдсэн нэг сувгаар фон процесс руу өгөгдөл нийлүүлдэг — \emph{Yahoo Finance} (\texttt{yfinance}-аар OHLCV ханш), \emph{AlphaVantage} (мэдээний сэтгэлгээний шинжилгээ), \emph{TradingView} (эдийн засгийн хуанли), \emph{Gemini API} (LLM-д суурилсан зах зээлийн шинжилгээ). \emph{Expo Push} нь зүүн-дээд буланд тусдаа байрласан бөгөөд сервер→мобайл чиглэлийн түлхэлтийн мэдэгдэл (push notification) хүргэх суваг (тасархай ягаан сум).

\textbf{Офлайн сургалт} (доод хэсэг, ашиглалтын орчноос тусдаа): түүхэн OHLCV (2015--2024) өгөгдөл дээр GBDT ансамблийг сургаж \texttt{model.pkl} артефакт үүсгэнэ. Энэ артефакт нь фон процесс руу \texttt{«байршуул»} сумаар хуулагдаж ашиглалтын таамаглалд хэрэглэгдэнэ. Гүйцэтгэлийн орчны нарийвчилсан UML байршуулалтын диаграммыг §3.8 (\fref{fig:deployment_view})-аас үзнэ үү.

\section{Өгөгдлийн урсгал}

Гадаад эх сурвалжаас (Yahoo Finance, AlphaVantage, TradingView) өгөгдөл орж ирээд, серверийн процессуудаар боловсруулагдан MongoDB цуглуулгад хадгалагдсаны дараа REST API-аар дамжин хэрэглэгчдэд хүрнэ. \subsection{Шошго (label) үүсгэх}

Сургалтын шошгыг ирээдүйн 240 минутын (4 цаг) ханшийн хөдөлгөөнд үндэслэн гурван ангилалд хуваасан:

\noindent\textit{Домэйны нэр томьёо}: Пипс (pip, percentage in point) нь Форекс зах зээлд ханшийн хамгийн бага өөрчлөлтийн нэгж; EUR/USD-д 1 пипс = 0.0001. Зах зээлийн хэлбэлзлийг тоолоход хэрэглэгддэг.

\begin{equation}
\text{label} = \begin{cases}
    \text{BUY (+1)} & \text{хэрэв } \Delta_{\text{up}} \geq 30\text{ пипс ба } \Delta_{\text{up}} > 1.5 \cdot \Delta_{\text{down}} \\
    \text{SELL ($-$1)} & \text{хэрэв } \Delta_{\text{down}} \geq 30\text{ пипс ба } \Delta_{\text{down}} > 1.5 \cdot \Delta_{\text{up}} \\
    \text{HOLD (0)} & \text{бусад тохиолдолд}
\end{cases}
\end{equation}

Үүнд $\Delta_{\text{up}}$ нь ирээдүйн 240 минутын дээд ханш ба одоогийн хаалтын зөрүү, $\Delta_{\text{down}}$ нь доод ханшийн зөрүү (пипсээр); 30 пипсийн босго нь EUR/USD-ийн дундаж өдрийн ATR(14) ($\approx$80 пипс)-ийн 37\%; 1.5 давамгайлалыг [1.2, 2.0] мужийг туршиж шошгын тэнцвэрийг харгалзан сонгосон.

Томьёоны бодит хэрэгжилтийг Алгоритм~\ref{alg:label_gen}-д псевдокод хэлбэрээр харуулав.

\begin{algorithm}[H]
\caption{Шошго үүсгэх алгоритм}
\label{alg:label_gen}
\algrenewcommand\algorithmicrequire{\textbf{Оролт:}}
\algrenewcommand\algorithmicensure{\textbf{Гаралт:}}
\begin{algorithmic}[1]
\REQUIRE OHLC цуваа $\{(t_i, o_i, h_i, l_i, c_i)\}_{i=1}^{N}$, ирээдүйн цонх $W=240$ мин, босго $\theta=30$ пип, давамгайлал $\rho=1.5$
\ENSURE Шошгын вектор $\mathbf{y} \in \{-1, 0, +1\}^N$
\STATE $\text{pip\_size} \gets 0.0001$ \quad // зөвхөн EUR/USD-д
\FOR{$i = 1, \ldots, N - W$}
    \STATE $H_{\max} \gets \max_{j \in [i+1,\, i+W]} h_j$
    \STATE $L_{\min} \gets \min_{j \in [i+1,\, i+W]} l_j$
    \STATE $\Delta_{\text{up}} \gets (H_{\max} - c_i) / \text{pip\_size}$
    \STATE $\Delta_{\text{down}} \gets (c_i - L_{\min}) / \text{pip\_size}$
    \STATE \textbf{if} $\Delta_{\text{up}} \geq \theta$ \textbf{and} $\Delta_{\text{up}} > \rho \cdot \Delta_{\text{down}}$ \textbf{then} $y_i \gets +1$ \quad // BUY
    \STATE \textbf{else if} $\Delta_{\text{down}} \geq \theta$ \textbf{and} $\Delta_{\text{down}} > \rho \cdot \Delta_{\text{up}}$ \textbf{then} $y_i \gets -1$ \quad // SELL
    \STATE \textbf{else} $y_i \gets 0$ \quad // HOLD
\ENDFOR
\STATE $y_{N-W+1}, \ldots, y_N \gets \text{NaN}$ \quad // сүүлийн $W$ candle-д шошго байхгүй
\STATE \textbf{буцаах:} $\mathbf{y}$
\end{algorithmic}
\end{algorithm}

\noindent\textbf{Нэвчилтийн хяналт.} Шинж чанарын цонх $[i-K+1,\,i]$ ба шошгын цонх $[i+1,\,i+W]$ давхцахгүй тул мэдээллийн нэвчилт алгоритмын түвшинд хязгаарлагдсан.

2015--2022 оны M1 сургалтын өгөгдөл (3{,}726{,}046 цэг) дээрх ангиллын хуваарилалт нь HOLD 75.5\% (2{,}812{,}515 цэг), BUY 12.2\% (454{,}999 цэг), SELL 12.3\% (458{,}532 цэг) хэлбэртэй гарсан. Энэхүү тэнцвэргүй байдлыг зохицуулахын тулд LightGBM ба XGBoost загваруудад \texttt{scale\_pos\_weight = neg/pos} параметрийг тохируулсан. Ангилал тус бүрийн загвар дээрх нөлөө болон BUY/SELL ангийн recall-ийн дүн шинжилгээг §4.2-т нэгтгэв.

\subsection{Шошго ба шинж чанарын цагийн нийцлийн аудит (leakage audit)}

Мэдээллийн нэвчилт гэдэг нь загварын сургалтад ирээдүйн мэдээлэл санамсаргүй нэвтрэх үзэгдэл бөгөөд энэ нь тестийн нарийвчлалыг хиймлээр өндөр харуулдаг. Шошго үүсгэх ба олон интервалын шинж чанар нэгтгэх үеийн гол шаардлага нь шийдвэрийн агшин ($t_0$)-д зөвхөн тухайн мөчөөс өмнө мэдэгдэх боломжтой мэдээлэл ашиглах явдал. Энэ нөхцөлийг хатуу хэлбэрээр томьёолбол:
\begin{equation}
t_{\text{feature}} \le t_0, \qquad \text{label window} = (t_0,\ t_0 + 240\text{ минут}]
\end{equation}

Шинж чанар үүсгэх ба шошго тооцох үе шатанд систем цагийн нийцлийн дүрмийг тогтвортой мөрддөг (томьёо 3.2). Бүх компонентын нарийвчилсан хяналтын дүрмийг Хавсралт~А-д тайлбарласан. \texttt{merge\_asof(backward)} болон хугацааны тусгаарлалтын дүрэм нь нэвчилтийн эрсдэлийг бууруулдаг ч олон интервалын синхрончлолын үлдэгдэл эрсдэлийг бүрэн үгүйсгэхгүй.

\subsection{WFV-ийн олон огтлолын алхамт логик}
\label{subsec:wfv_logic}

Нийт \textbf{5 fold} ашигласан бөгөөд train cutoff жил бүр 1 жилээр шилжсэн (Fold 1: 2018, Fold 2: 2019, $\ldots$, Fold 5: 2022); сургалтын цонх 4→8 жил болж өргөжсөн ба баталгаажуулалт (val) ба тест тус бүрд 1 жилийн дараалсан цонх ашигласан (Fold $k$-ийн val жил = train cutoff + 1, test жил = train cutoff + 2). Энэ нь \textbf{step size = 1 жил} бүхий expanding window протокол юм. \section{AI загварын бүтэц ба сургалт}

\subsection{Ансамбль загварын архитектур}

Системийн загвар нь гурван GBDT (gradient boosted decision trees) загвараас бүрдсэн ансамбль бөгөөд олон модны нэгтгэсэн архитектураар санхүүгийн хүснэгтэн өгөгдөлтэй сайн нийцдэг \cite{chen2016xgboost}. Загвар бүр ижил 48 шинж чанарыг хүлээн авч BUY, SELL, HOLD гурван ангиллын магадлалыг тус тус тооцоолно.

\subsubsection{Загварын математик тодорхойлолт}
\label{sec:math_formulation}

Сургалтын өгөгдөл $\mathcal{D} = \{(\mathbf{x}_i, y_i)\}_{i=1}^{N}$, $\mathbf{x}_i \in \mathbb{R}^{48}$,
$y_i \in \{0, 1, 2\}$ (HOLD=0, BUY=1, SELL=2). Суурь загвар $k \in \{\text{LGB, XGB, CAT}\}$ нь
давталтын \emph{gradient boosting} аргачлалаар $T$ шийдвэрийн модны нийлбэрийг сурдаг:

\begin{equation}
F_k^{(T)}(\mathbf{x}) = \sum_{t=1}^{T} \eta \cdot f_t(\mathbf{x}), \qquad f_t \in \mathcal{F}
\label{eq:gbdt_sum}
\end{equation}

\noindent Үүнд $\mathcal{F}$ нь шийдвэрийн модны функцын орон зай, $\eta=0.03$ нь
learning rate, $f_t$ нь $t$-р алхамын үед өмнөх ансамблийн \emph{градиентыг
бууруулахаар} сурсан мод юм:

\begin{equation}
f_t = \arg\min_{f \in \mathcal{F}} \sum_{i=1}^{N} \ell\bigl(y_i,\, F_k^{(t-1)}(\mathbf{x}_i) + f(\mathbf{x}_i)\bigr) + \Omega(f)
\label{eq:boosting_step}
\end{equation}

\noindent энд $\Omega(f)$ нь модны нарийн төвөгшилт хариуцсан тогтмолжуулалт.
Гурван анги тул логит $\mathbf{z} \in \mathbb{R}^3$-ыг softmax-аар магадлал болгон хувиргана:

\begin{equation}
P_k(y = c \mid \mathbf{x}) = \frac{\exp(z_c)}{\sum_{c'=0}^{2} \exp(z_{c'})},
\qquad \mathbf{z} = F_k(\mathbf{x})
\label{eq:softmax}
\end{equation}

\subsubsection{Алдагдлын функц (loss function)}
\label{sec:loss_function}

Гурван загвар бүгд \textbf{multi-class cross-entropy} (мөн \emph{softmax loss}, \emph{log-loss}
гэж нэрлэдэг) алдагдлын функцыг минимумжуулдаг:

\begin{equation}
\boxed{
\mathcal{L}(\mathcal{D}; \theta_k) = -\frac{1}{N} \sum_{i=1}^{N} \sum_{c=0}^{2}
\mathbb{1}[y_i = c] \cdot \log P_k(y_i = c \mid \mathbf{x}_i; \theta_k)
+ \lambda_1 \|\theta_k\|_1 + \lambda_2 \|\theta_k\|_2^2
}
\label{eq:loss}
\end{equation}

\noindent Үүнд $\theta_k$ нь $k$-р загварын бүх параметрийн вектор,
$\mathbb{1}[\cdot]$ нь индикатор функц (зөв ангилалд $1$, бусдад $0$),
$\lambda_1, \lambda_2$ нь L1 ба L2 тогтмолжуулалтын хүчний коэффициент
(\tref{tab:hyperparams_full}-аас LightGBM ба XGBoost-д $\lambda_1 = 0.10,
\lambda_2 = 1.0$; CatBoost-д L2 коэффициент ($\texttt{l2\_leaf\_reg}$) $3.0$).
HOLD ангиллын давамгайллаас үүсэх тэнцвэргүй байдлыг зохицуулахын тулд
LightGBM ба XGBoost загваруудад \texttt{scale\_pos\_weight} параметрийг сөрөг
ба эерэг ангийн харьцаагаар ($\text{neg}/\text{pos}$) тохируулсан.

\subsubsection{Ансамблийн нэгтгэх дүрэм}

Гурван загварын магадлалыг \textbf{зөөлөн санал хураалтаар}
тэгш жингээр дундажилж эцсийн магадлалыг гаргана:

\begin{equation}
P_{\text{ensemble}}(c \mid \mathbf{x}) = \sum_{k=1}^{3} w_k \cdot P_k(c \mid \mathbf{x}),
\qquad w_k = \tfrac{1}{3}
\label{eq:soft_voting}
\end{equation}

Ангилалын шийдвэр болон итгэлцлийг доорх томьёогоор тодорхойлно:

\begin{equation}
\hat{c} = \arg\max_{c}\, P_{\text{ensemble}}(c \mid \mathbf{x}), \qquad
\text{conf} = \max_{c}\, P_{\text{ensemble}}(c \mid \mathbf{x})
\label{eq:decision_rule}
\end{equation}

\noindent Үүнд $\hat{c}$ нь сонгосон ангилал, $\text{conf} \in [\tfrac13, 1]$ нь ансамблийн итгэлцэл болно. \fref{fig:ensemble_arch} нь энэхүү бүтцийг харуулав.

Ансамблийн жинг $w=(\tfrac13,\tfrac13,\tfrac13)$ тэгш авсан шалтгаан нь нэг огтлолд жинг оновчлох замаар үүсэх data-snooping эрсдэлээс зайлсхийх зорилготой. HOLD ангилал сонгогдсон үед систем шинэ байрлал нээхгүй (no-trade state).

\subsection{Гиперпараметр сонголтын протокол}

Гиперпараметрийг grid search-ийн оронд тогтвортой байдлын протоколоор сонгосон: train-val зөрүү $<5\%$, conf~$\geq\!0.90$ дэд мужийн нарийвчлал $>90\%$, шинж чанарын top-10 жагсаалтын Jaccard коэффициент $>0.7$ гэсэн гурван шалгуурыг зэрэг хангах утгуудыг хайсан. Ансамбль сургалтыг \texttt{random\_state=42} нэг seed-ийн протоколоор явуулсан. Гол сонголтууд: $n_{\text{estimators}}\!=\!500$, $\eta\!=\!0.03$, \texttt{max\_depth}~$\!\in\!\{5,6\}$, \texttt{reg\_lambda}~$\!\in\!\{1.0, 3.0\}$, \texttt{early\_stopping\_rounds}~$=\!50$. Хэт нийцлээс сэргийлэх зургаан арга хэмжээ (модны гүнийг хязгаарлах, бага learning rate, эрт зогсоолт, L1/L2 нормчлол, WFV, шинж чанарын цөөрөлт) бүгд хослуулан хэрэгжүүлсэн. Гурван загварын бүх гиперпараметрийн нарийвчилсан утгуудыг ялгаа болон үүргийн тайлбартай нь \textbf{Хавсралт~Е}-д (\tref{tab:hyperparams_full}) нэгтгэв.

\subsection{Дохионы шүүлтүүр ба тоон үр дүн}
\label{sec:signal_filter}

Процесс P2 (\fref{fig:data_flow_unified}) дохиог хоёр нөхцлөөр шүүгээд D1 (\texttt{signals}) агуулахад хадгална: (1)~ансамблийн таамагласан анги BUY эсвэл SELL байх (HOLD биш) \textbf{ба} (2)~M1 ATR $\geq 4$ пипс. Автоматаар үүсгэх мөчлөг нь итгэлцлийн босгыг энэ шатанд хэрэглэдэггүй; дээрх хоёр нөхцлийг хангасан дохио бүрийг upsert үйлдлээр MongoDB-д хадгална. Итгэлцлийн босго (анхдагчаар $0.90$) нь дараа нь Expo Push мэдэгдлийг хэрэглэгчид илгээх шатанд, хэрэглэгч бүрийн хувийн тохиргооны дагуу хэрэглэгдэнэ. ATR шүүлтүүр нь зах зээлийн идэвхгүй хугацаанд (шөнийн хаагдалт, долоо хоногийн эцэс) буурсан хэлбэлзлийн үед үүсэх сул дохиог хасдаг. 2025 оны бодит өгөгдөл дээрх тоон үр дүн (стратегийн гүйцэтгэлийн дэлгэрэнгүй үнэлгээг §\ref{ch:results}-аас үзнэ үү): нийт 359{,}639 таамгаас итгэлцэл $\geq\!0.90$ ба ATR $\geq\!4$ пипсийн нөхцөлд 3{,}567 (1.0\%) дохио тэнцсэн. \subsection{Stop Loss ба Take Profit тооцоолол}

Арилжааны эрсдэлийг хязгаарлах Stop Loss (SL) болон ашгийг хамгаалах Take Profit (TP) утгуудыг ATR-д суурилан динамикаар тооцоолсон:

\begin{equation}
    \text{SL} = \max(\text{ATR}_{14} \times 5.0,\ 15\text{ пипс})
\end{equation}
\begin{equation}
    \text{TP} = \max(\text{SL} \times 3.0,\ 45\text{ пипс})
\end{equation}

ATR үржвэр 5.0 нь 7 давталтын туршилтаар 3.0--7.0 мужаас тогтвортой гүйцэтгэл өгсөн утга; TP/SL=3:1 нь ердөө 33\%-ийн ялалтын хувьд ч санхүүгийн хувьд ашигтай байх математик нөхцөл; 15/45 пипсийн доод хязгаар нь санамсаргүй хөдөлгөөн болон зардлаас хамгаалдаг.

Босгын мужийн хүснэгт нь ажиллах үеийн динамик тааруулалт бус, \textbf{сценарийн харьцуулалт} бөгөөд 0.91 нь энэ шинжилгээний хүрээнд ажиглагдсан локал дээд цэг тул бүх нөхцөлд универсал оновчтой утга гэж дүгнэхгүй.

TP/SL=3:1 нөхцөлд $E[R] = p_{\text{win}} \cdot 3 - (1 - p_{\text{win}})$ томьёогоор Expectancy тооцно; $E[R] > 0$ бол статистикийн давуу талтай.

\noindent\textit{Эрсдэлийн үзүүлэлтүүдийн тодорхойлолт}: MaxDD (Maximum Drawdown) нь балансын оргил цэгээс хамгийн гүн уналтын зай. Sharpe нь илүүдэл өгөөжийг нийт хэлбэлзлээр хуваасан харьцаа; Sortino нь зөвхөн доошлох хэлбэлзлийг (downside volatility) тооцдог Sharpe-ын хувилбар; Calmar нь жилийн өгөөжийг MaxDD-д хуваасан харьцаа. Win rate-ийн 95\% итгэлцлийн интервалыг Wilson аргаар (хоёртын хувь хэмжигдэхүүнд тохирсон, жижиг түүвэрт нарийвчлалтай тооцоолол) тооцов.

\section{Харагдах хэсгийн архитектур}
\label{sec:modular_api_blueprint}

Харагдах хэсгийн зорилго нь серверийн хэсгээс ирэх ханш, дохио, мэдээ, мэдэгдлийг нэгтгэн хэрэглэгчид тасралтгүй хүргэхэд оршино; мобайл давхаргыг нэвтрэлт, өгөгдөл харуулах, сервертэй синхрончлол гэсэн гурван түвшний модулиар хэрэгжүүлсэн.

\subsection{Харагдах хэсгийн модулийн бүтэц ба API харилцаа}

Харагдах хэсэг нь нэвтрэлт, ханш, дохио, мэдээ гэсэн дөрвөн бие даасан модулиас бүрдэнэ. Модуль тус бүрийн үүрэг, харилцааны загвар (REST GET polling эсвэл POST/PUT) болон холбогдох API endpoint-ийн бүрэн жагсаалтыг Хавсралт~Ж-ээс үзнэ үү.
