\subsection*{WFV олон огтлолт (expanding window) протокол}

Урагш гүйлгэн баталгаажуулалтыг цаг хугацааны дараалал хадгалсан \textbf{олон огтлолт, өргөтгөх сургалтын цонх} (expanding window) протоколоор хэрэгжүүлсэн. Өөрөөр хэлбэл Fold 1--Fold 5 огтлол бүрт сургалт\(\to\)баталгаажуулалт\(\to\)тестийн алхмыг урагшлуулан давтаж, ирээдүйн мэдээлэл сургалтад нэвтрэх эрсдэлийг хязгаарласан.

\noindent\textbf{Олон огтлолтын ерөнхий алхамт логик:}
\begin{enumerate}
    \item Эхний хугацааны огтлолд өмнөх жилүүдээр загвар сургана (Fold 1: 2015--2017)
    \item Дараагийн хугацааны огтлолд баталгаажуулалт/шалгалт (early stopping, calibration) хийнэ
    \item Дараагийн огтлол руу шилжихдээ сургалтын цонхыг урагшлуулж дахин сургана (Fold 2: 2015--2018, ..., Fold 5: 2015--2022)
    \item Эцсийн үнэлгээнд сүүлийн огтлолын (Fold 5) түүврийн гадуурх (out-of-sample) гүйцэтгэлийг ашиглана (2024 тестийн өгөгдөл)
\end{enumerate}

\subsection*{Хэмжилтийн нөхцөл}

API давхаргын p95 latency хэмжилтийг дараах нөхцөлд явуулсан:
\begin{itemize}
    \item \textbf{Эх цэг}: Улаанбаатар (Mongolia)
    \item \textbf{Серверийн байршил}: Токио (Tokyo, Fly.io)
    \item \textbf{Сүлжээний RTT}: дунджаа 150--180ms
    \item \textbf{Хүсэлтийн тоо}: Эндпойнт тутамд n=30 хүсэлт
    \item \textbf{Ачааллын загвар}: Дараалсан (sequential) ачаалал, өөр өөр 3 endpoint-т
    \item \textbf{Хэмжилтийн хугацаа}: 15 минут (900 сек), тасралтгүй
\end{itemize}

Worker ба Push урсгалын KPI:
\begin{itemize}
    \item \textbf{Worker job completion}: Өдөрт 50--100 ажил процесс хийгдэж, лог дээр бичигдэнэ
    \item \textbf{Worker scheduling}: Python threading (time.sleep мөчлөгт суурилсан) ашиглан минут бүр дохио үүсгэх мөчлөгийг гүйцэтгэдэг
    \item \textbf{Push delivery}: Firebase FCM платформ, хүргэлтийн статистик FCM console-ээр тайлагнана
\end{itemize}

\subsection*{p95 latency зорилтот босгоос хэтрэлтийн дүн шинжилгээ}

API давхаргын p95 latency 471--536ms байсан нь зорилтот 350ms-ийг хэтэрсэн. Үндсэн шалтгааны дүн шинжилгээ:

\begin{enumerate}
    \item \textbf{Сүлжээний RTT нөлөө}: Улаанбаатараас Токиогийн сервер хүртэлх RTT дунджаа 150--180ms байдаг тул сүлжээний холболтын орыг зохисон хүлээлтэнд оруулах шаардлагатай
    \item \textbf{MongoDB query latency}: /signals/latest endpoint-т MongoDB дээр дохиог авах query хэрэгтэй бөгөөд үндсэн database-д индекс оновчлол хийхээр үлдсэн
    \item \textbf{Сериалчлалын ачаалал}: JSON сериалчлалын үед томоохон объектын цуглуулгыг үйлдэх үйлүүлэгдсэн
    \item \textbf{Кэшлэлтийн хомсдол}: /rates/live-ийн үр дүнг Redis кэш-д байнаар хадгалснаас гадна, түүвэр дүүргэлтийн үедээ кэш сайн ажиллавал бууна
\end{enumerate}

Практик сайжруулах сонголтууд:
\begin{itemize}
    \item MongoDB дохионы хадгалалтад \texttt{(timestamp, symbol, timeframe)} compound unique index + \texttt{upsert} бодлого хэрэгжүүлэх (idempotency)
    \item Серверийн бүсчлэлийг хэрэглэгчийн байршилд ойртуулах, эсвэл CDN/edge cache ашиглах
    \item Redis multi-tier кэшлэл (L1 in-memory, L2 Redis)
    \item Мейл query-ийн цуврал хүсэлт (batch requests) хийх
\end{itemize}

\subsection*{Дахин хэрэгжүүлэх боломж ба орчны нөхцөл}

Энэ бүлгийн үр дүнг дахин хэрэгжүүлэхэд дараах нөхцөлүүдийг тогтмол хадгалах шаардлагатай:

\begin{itemize}
    \item \textbf{Эх сурвалжийн лог багц}: Босгын мужийн 0.90--0.99 бүх гүйлтийн MT5 логийг өөрчлөхгүй хадгалах
    \item \textbf{Нэгтгэлийн дүрэм}: Босго бүрт \texttt{Loaded signals} anchor мөрөөс хойших эхний \texttt{final balance} мөрийг холбох
    \item \textbf{Арилжааны үзүүлэлтүүд}: \texttt{Trades/Trade} мөрөөс TP/SL болон хаагдсан арилжааны дарааллыг нэг ижил дүрмээр тооцох
    \item \textbf{Гүйцэтгэлийн нөхцөл}: Ижил MT5 тестийн хугацааны цонх, ижил EA ажиллах үеийн дүрмүүд (нэг агшинд нэг байрлал, ижил дохио ачаалах журам)
\end{itemize}

Эдгээр нөхцөл зөрвөл тайланд буй тоон үр дүнтэй 1:1 давхцахгүй байж болох тул дахин хэрэгжүүлэлтийн үед параметрийн өөрчлөлт бүрийг тусдаа протоколын хүснэгтээр бүртгэх нь зүйтэй.

\subsection*{Практик шийдвэр гаргалтын нэмэлт тайлбар}

Практик шийдвэр гаргалтын хүрээний дэлгэрэнгүй тайлбарыг Үр дүнгийн бүлгийн ``Практик шийдвэр гаргалтын хүрээ'' хэсгээс үзнэ үү. Тухайн хэсэгт эрсдэл-өмнөх шалгуур (CV, MaxDD, урт дараалсан алдагдал, win rate CI) болон босгын практик мужийн нөхцөлт тайлбарыг нэгтгэн харуулсан болно.
