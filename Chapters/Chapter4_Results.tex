\section{Үр дүнгийн өгөгдлийн эх сурвалж}

\section{Загварын ангилалтын гүйцэтгэл}
\label{sec:model_classification_performance}

Судалгааны асуулт~1 (RQ1): ``Загвар нь цагийн дарааллыг алдагдуулахгүй тест дээр тогтвортой, найдвартай ажиллах уу?''~гэсэн асуултад хариулахын тулд GBDT ансамблийн ангилалтын нарийвчлалыг Walk-Forward Validation (WFV) схемийн дагуу тусдаа өгөгдлийн хуваарь дээр хэмжив. \subsection{Дохионы гүйцэтгэлийн чанар}

Түүхэн өгөгдөл дээрх системийн туршилтын логийг босго тус бүрээр задлахад \textbf{шүүгдсэн дохионы тоо} болон \textbf{бодитоор нээгдэж хаагдсан арилжааны тоо} хооронд тогтвортой зөрүү ажиглагдсан. Энэ зөрүү нь дараах гүйцэтгэлийн механизмтай холбоотой:
\begin{itemize}
    \item Нэг агшинд зөвхөн нэг байрлал нээх дүрэм (MaxPositions=1)
    \item Нээлттэй байрлал хаагдахаас өмнөх давхцсан дохионуудыг алгасах
    \item Зөвхөн тухайн хэрэгсэл дээрх дохиог гүйцэтгэх хязгаарлалт
\end{itemize}

Иймд гүйцэтгэлийн хувийг \(\text{Exec.}(\%) = \frac{\text{Арилжаа}}{\text{Ачаалагдсан дохио}} \times 100\) томьёогоор тооцсон (хүснэгтэд ``Loaded'' гэж богиносгов). Гарсан 6.45--23.81\%-ийн гүйцэтгэлийн хувь нь системийн алдаа бус, \textbf{зориудын чанарын шүүлтүүрийн үр дүн} юм — загвар минут бүр таамаглал гаргадаг ч зөвхөн итгэлцлийн босго болон MaxPositions=1 дүрмийг\footnote{MaxPositions=1: нэг агшинд зөвхөн нэг нээлттэй байрлал (open position) байхыг шаардах хязгаарлалт. ``Байрлал'' (position) гэдэг нь арилжаа нээгдэж, хаагдаагүй байгаа төлвийг хэлнэ.} давсан дохио л арилжаа болдог. Break-even ялалтын хувь 25\% байна. Босго өсөх тусам арилжааны тоо буурч, итгэлцлийн интервал өргөсдөг.

Эдгээр нь хэлцлийн логийн өдрийн баланс дараалалд суурилсан хэмжилт бөгөөд арилжааны тоо цөөн (35--230 хэлцэл), хугацааны цонх богино (1 жил), зардлын бүрэн хамралтгүй нөхцөлд гарсан. Иймд эдгээрийг зөвхөн \textbf{босгуудын харьцангуй харьцуулалтын суурь} болгон авч үзэх бөгөөд боломжит хөрөнгө оруулалтын шийдвэрийн үндэс болгохгүй.

Энэ графикаас харахад 0.93 (9.53\%) болон 0.98 (7.66\%) босгон дээр уналтын эрсдэл харьцангуй өндөр байна. Харин 0.99 дээр MaxDD хамгийн бага (1.98\%) боловч нийт өгөөж болон дохионы тоо мэдэгдэхүйц буурсан. Иймээс босго сонгохдоо зөвхөн өгөөж бус, уналтын эрсдэлийг хамтад нь авч үзэх шаардлагатай.

\sloppy

Баталгаажуулалтын протокол (Methodology-д тодорхойлсон) нь API давхарга, worker урсгал, push урсгалын хэмжигдэх шалгуурыг тогтоосон. Энэ хэсэгт тухайн протоколын дагуу явуулсан туршилтын бодит дүнг тайлагнана. Хэмжилтийн нөхцөл, методология, болон дэлгэрэнгүй үнэлгээний дүнг Хавсралт~Г-аас үзнэ үү.

\noindent\textit{Тайлбар:} 90 хүсэлтийн аль нэг нь ч 5xx кодтой хариу өгөөгүй. HTTP 200 хариугийн хувь 100\% байв.

API давхаргын p95 latency 471--536ms байсан нь зорилтот 350ms-ийг хэтэрсэн. Гол шалтгаан нь хэмжилтийн эх цэг (Улаанбаатар) ба сервер (Токио) хоорондын RTT юм. Нийт хоцролтын задаргааг дараах байдлаар тайлбарлана:

\begin{itemize}
    \item \textbf{Сүлжээний RTT} (Улаанбаатар $\to$ Токио): $\approx$150--180\,ms
    \item \textbf{Серверийн боловсруулалт} (Flask + MongoDB query): $\approx$300\,ms
    \item \textbf{Нийт p95} $\approx$ 471\,ms \checkmark
\end{itemize}

\noindent Иймээс p95 latency зорилт биелээгүй шалтгааныг зөвхөн сүлжээний гадаад нөлөө гэж үзэхгүй; серверийн байршлыг Токио болгон сонгосон архитектурын шийдвэр өөрөө энэ зөрүүнд шууд нөлөөлсөн. Энэ нь \textbf{архитектурын гүйцэтгэлийн зөрүү} бөгөөд дараах гурван чиглэлд шийдвэрлэх боломжтой: (1) серверийг хэрэглэгчийн бүсэд (Улаанбаатар, Зүүн Ази) ойртуулах бүсчлэл (regional deploy), (2) `/rates/live` зэрэг байнга хандагддаг endpoint-д CDN/edge cache нэмэх, (3) MongoDB query-г индекс оновчлол болон холболтын санг (connection pooling) сайжруулах замаар $\approx$300\,ms-ийн боловсруулалтын хугацааг бууруулах. Эдгээр шийдлийг хэрэгжүүлсний дараа p95 зорилтыг хангах нь техникийн хувьд боломжтой гэж үзэж байна.

\subsubsection{Worker болон Push үйлчилгээний архитектурын байдал}

Worker урсгал нь Python threading-д суурилсан бөгөөд минут бүр өгөгдөл татаж, GBDT ансамблиар таамаглал хийж, MongoDB-д хадгалах мөчлөгийг автоматаар гүйцэтгэдэг. Firebase FCM-ээр дамжуулан шинэ дохио үүсэх бүрт мобайл апп руу push мэдэгдэл илгээгддэг. Эдгээр бүрэлдэхүүнүүд прототипийн орчинд тогтвортой ажиллаж байгаа боловч бүтээмжийн хэмжилтийн (Prometheus, structured log counter) нэгдсэн дэд бүтэц дараагийн шатны инженерчлэлийн ажилд хамаарна.

\section{Дахин хэрэгжүүлэх боломж ба орчны нөхцөл}

Энэ бүлгийн үр дүнг дахин хэрэгжүүлэхэд дараах нөхцөлүүдийг тогтмол хадгалах шаардлагатай: MT5 логийн нэгтгэлийн дүрэм (\texttt{(timestamp, symbol)} гэсэн compound index), дохио бүрийн \texttt{model\_version} ба \texttt{run\_id} ул мөр, мөн гүйцэтгэлийн EA тохиргоо (\tref{tab:ea_params}-ийн параметрүүд). Дохионы давхардлаас сэргийлэхийн тулд MongoDB-д \texttt{upsert=True} болон compound unique index хэрэгжүүлсэн бөгөөд энэ нь ижил дохиог дахин бичих эрсдэлийг прототипийн түвшинд бууруулсан. Дэлгэрэнгүй нөхцөл ба протоколын жагсаалтыг Хавсралт~Д-аас үзнэ үү.
